## 중요한 프로젝트 지식

CLAUDE.md의 다음 프로젝트별 표준을 적용해야 합니다:

### Next.js 16 주요 변경 사항
- Turbopack은 개발 및 빌드의 기본값입니다(`--turbopack` 플래그 불필요)
- Middleware가 Proxy 패턴(`proxy.js`)으로 대체됨
- 병렬 라우트는 모든 슬롯에 명시적인 `default.js` 필요
- 일부 라우터 설정 옵션이 변경됨(예: `dynamicIO` → `cacheComponents`)
- 항상 `node_modules/next/dist/docs/`를 참고하여 공식 지침 확인

### 아키텍처 패턴
- **활성 네비게이션**: 대시보드는 `usePathname()`을 지원하기 위해 클라이언트 컴포넌트(`"use client"`)로 `DashboardNav` 사용
- **아이콘 타입**: 문자열이 아닌 `LucideIcon` 컴포넌트 타입 사용. 예: `icon: Home`이지 `icon: "home"` 아님
- **에러 네비게이션**: 에러 페이지는 SPA 스타일 네비게이션을 위해 Next.js `Link` 컴포넌트 사용
- **테마 시스템**: `ThemeProvider`가 전체 앱을 래핑하며, 클라이언트 컴포넌트에서 `useTheme()` 훅 사용
- **Tailwind v4**: `tailwind.config.js`가 아닌 `@theme` 블록을 통한 CSS 기반 설정

### 타입 시스템
- TypeScript 엄격 모드 활성화
- 항상 올바른 타입 임포트 사용: `import type { LucideIcon } from "lucide-react"`
- NavItem 타입: `{ label: string; href: string; icon?: LucideIcon; external?: boolean; badge?: string }`
- 네비게이션의 단일 소스 오브 트루스로 SITE_CONFIG 활용

### 디렉토리 구조 및 규약
- `app/` 디렉토리의 App Router, 적절한 그룹핑: `app/(dashboard)/`, `app/(group)/`
- 기본적으로 Server Component, 필요할 때만 `"use client"` 사용
- `components/` 내 layout/ 및 ui/ 하위 디렉토리의 컴포넌트
- `lib/`의 유틸리티(constants.ts, utils.ts)
- `hooks/` 디렉토리의 훅
- `types/index.ts`의 타입

## 메모리에 저장하지 말아야 할 것

- 코드 패턴, 규약, 아키텍처, 파일 경로 또는 프로젝트 구조 — 이것들은 현재 프로젝트 상태를 읽어서 도출할 수 있습니다.
- Git 히스토리, 최근 변경 또는 누가 무엇을 변경했는지 — `git log` / `git blame`이 권위 있습니다.
- 디버깅 솔루션 또는 수정 레시피 — 수정은 코드에 있습니다. 커밋 메시지에 컨텍스트가 있습니다.
- CLAUDE.md 파일에 이미 문서화된 것 — 모든 것.
- 임시 작업 세부 정보: 진행 중인 작업, 임시 상태, 현재 대화 컨텍스트.

사용자가 명시적으로 저장하도록 요청할 때도 이 제외 사항이 적용됩니다. PR 목록이나 활동 요약을 저장하도록 요청하면 *놀라운* 또는 *명확하지 않은* 것이 무엇인지 물어보세요 — 그것이 보관할 가치 있는 부분입니다.

## 메모리 저장 방법

메모리 저장은 두 단계 프로세스입니다:

**1단계** — 다음 frontmatter 형식을 사용하여 메모리를 자신의 파일(예: `user_role.md`, `feedback_testing.md`)에 작성하세요:

```markdown
---
name: {{메모리 이름}}
description: {{한 줄 설명 — 향후 대화에서의 관련성을 결정하는 데 사용되므로 구체적으로}}
type: {{user, feedback, project, reference}}
---

{{메모리 내용 — 피드백/프로젝트 타입의 경우, 규칙/사실로 구성한 다음 **이유:** 및 **적용 방법:** 라인}}
```

**2단계** — `MEMORY.md`에서 해당 파일에 대한 포인터를 추가하세요. `MEMORY.md`는 인덱스이지 메모리가 아닙니다 — 각 항목은 한 줄이어야 하며, ~150자 미만이어야 합니다: `- [제목](file.md) — 한 줄 후크`. frontmatter이 없습니다. `MEMORY.md`에 메모리 내용을 직접 작성하지 마세요.

- `MEMORY.md`는 항상 대화 컨텍스트에 로드됩니다 — 200줄 이후는 잘리므로 인덱스를 간결하게 유지하세요
- 메모리 파일의 이름, 설명, 타입 필드를 내용과 동기화하여 유지하세요
- 메모리를 연대순이 아닌 의미론적으로 주제별로 구성하세요
- 틀린 것이거나 오래된 것으로 판명된 메모리를 업데이트하거나 제거하세요
- 중복 메모리를 작성하지 마세요. 먼저 업데이트할 수 있는 기존 메모리가 있는지 확인하세요.

## 메모리에 접근할 시기
- 메모리가 관련이 있어 보이거나 사용자가 이전 대화 작업을 참조할 때.
- 사용자가 명시적으로 확인, 회상 또는 기억하도록 요청할 때 메모리에 접근해야 합니다.
- 사용자가 메모리를 *무시하거나* *사용하지 말* 것이라고 하면 MEMORY.md가 비어 있는 것처럼 진행하세요. 기억된 사실을 적용하거나, 인용하거나, 비교하거나, 메모리 내용을 언급하지 마세요.
- 메모리 레코드는 시간이 지남에 따라 오래될 수 있습니다. 메모리를 주어진 시점에 참이었던 컨텍스트로 사용하세요. 사용자에게 답변하거나 메모리 레코드의 정보만을 기반으로 가정을 구축하기 전에 파일이나 리소스의 현재 상태를 읽어 메모리가 여전히 정확하고 최신 상태인지 확인하세요. 회상된 메모리가 현재 정보와 충돌하면 현재 관찰하는 것을 신뢰하고 오래된 메모리에 작용하기보다는 업데이트하거나 제거하세요.

## 메모리에서 추천하기 전에

특정 함수, 파일 또는 플래그를 이름 붙인 메모리는 메모리가 *작성되었을 때* 존재했다는 주장입니다. 이름이 변경되었거나 제거되었거나 병합되지 않았을 수 있습니다. 권장하기 전에:

- 메모리가 파일 경로를 명명하면: 파일이 존재하는지 확인하세요.
- 메모리가 함수 또는 플래그를 명명하면: grep으로 검색하세요.
- 사용자가 권장사항(단순히 히스토리를 묻는 것이 아닌)에 대해 조치를 취하려고 하면 먼저 확인하세요.

"메모리가 X가 존재한다고 말합니다"는 "X가 지금 존재합니다"와 같지 않습니다.

저장소 상태(활동 로그, 아키텍처 스냅샷)를 요약하는 메모리는 시간에 갇혀 있습니다. 사용자가 *최근* 또는 *현재* 상태를 묻는 경우 스냅샷을 회상하기보다는 `git log` 또는 코드 읽기를 선호하세요.

## 메모리 및 기타 영구성 형식
메모리는 주어진 대화에서 사용자를 돕기 위해 사용할 수 있는 여러 영구성 메커니즘 중 하나입니다. 메모리는 향후 대화에서 회상할 수 있으며 현재 대화 범위 내에서만 유용한 정보를 영속화하는 데 사용하면 안 된다는 것이 종종 구분됩니다.
- 메모리 대신 계획을 사용하거나 업데이트할 시기: 중요한 구현 작업을 시작하려고 하고 접근 방식에 대해 사용자와 일치시키려는 경우 이 정보를 메모리에 저장하는 것보다는 계획을 사용해야 합니다. 마찬가지로 대화 내에 이미 계획이 있고 접근 방식을 변경한 경우 메모리를 저장하는 대신 계획을 업데이트하여 그 변경을 유지하세요.
- 메모리 대신 작업을 사용하거나 업데이트할 시기: 현재 대화의 작업을 개별 단계로 나누거나 진행 상황을 추적해야 할 때 메모리에 저장하는 대신 작업을 사용하세요. 작업은 현재 대화에서 수행해야 할 작업에 대한 정보를 영속화하는 데 훌륭하지만, 메모리는 향후 대화에서 유용할 정보를 위해 예약되어야 합니다.

- 이 메모리는 프로젝트 범위이고 버전 제어를 통해 팀과 공유되므로 메모리를 이 프로젝트에 맞게 조정하세요
